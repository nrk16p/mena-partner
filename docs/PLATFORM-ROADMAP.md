# MENA Partner Platform — แผนแม่บทระบบ (Platform Roadmap)

> จัดทำ: 5 ก.ค. 2026 · สถานะ: ฉบับเสนอทีม (รอ review)
> เป้าหมาย: ยกระดับ mena-partner จาก "รวมหน้าจอ" เป็น **ระบบหลักของธุรกิจรถร่วม Mixer ครบวงจร**
> ตั้งแต่ทำสัญญา → ข้อมูลหลักรถ/คนขับ → เที่ยววิ่ง → เงินเดือน → รายการหักทุกประเภท → ปิดเดือน → รายงาน

---

## 1. วิสัยทัศน์

**"ข้อมูลทุกบาทของรถร่วม 1 คัน อยู่ในระบบเดียว เห็นตรงกันทุกหน้า ตรวจสอบย้อนหลังได้"**

หลักการออกแบบ 3 ข้อ (บังคับใช้กับทุกฟีเจอร์ใหม่):
1. **Single Source of Truth** — ตัวเลขเดียวกันต้องมาจากแหล่งเดียว (แบบที่ทำกับ `lib/promo-usage.ts` และ `lib/payroll-engine.ts`)
2. **Master data ก่อน transaction** — รถ/คนขับ/สัญญา ต้องสมบูรณ์ก่อน จึงบันทึกรายการเงินได้
3. **ปิดเดือนแล้วล็อก** — ข้อมูลที่จ่ายเงินแล้วแก้ไม่ได้ (audit trail)

## 3. Core Data Model — ชั้นข้อมูล 3 ชั้น

```
[MASTER]      drivers ── contracts ── vehicle_master ── master_price_list ── promo_config
                              │ (contractCode = กุญแจหลัก, licensePlate = กุญแจรอง)
[TRANSACTION] trips · stock_movements · repair_claims · pm_records · debt_acceptances
              fuel_records · monthly_adjustments · installment_schedule · (ใหม่: incident_cases)
                              │
[DERIVED]     lib/payroll-engine → payroll_entries (ห้ามแก้มือ)
              lib/promo-usage    → ยอดใช้โปรฯ (ห้ามเก็บซ้ำ)
              repair_monthly     → สรุปซ่อมรายเดือน
```

**กฎเหล็ก:** ชั้น DERIVED ต้องคำนวณจากชั้นล่างเสมอ ห้าม UI เขียนตัวเลขสรุปเอง
(บั๊กที่แก้ไป 5 ก.ค. เกิดจากฝ่าฝืนข้อนี้ — ยอดใช้โปรฯ มี 3 นิยามใน 3 ที่)

### จุดที่ต้อง normalize ให้เป็นมาตรฐานเดียว
- **ทะเบียนรถ**: เก็บ raw `สบ.71-1959` + เทียบด้วย `normPlate()` → `71-1959` (มีแล้วใน lib/promo-usage)
- **contractCode**: กุญแจเชื่อมทุก collection — ตอนนี้ mapping สมบูรณ์ผ่าน repair_monthly (102 คู่)
- **วันที่**: string `YYYY-MM-DD` ทั้งหมด + เดือนบัญชี `YYYY-MM`

## 4. Workflow หลักของทีม (End-to-End)

### 4.1 Onboarding รถร่วมใหม่ (ครั้งเดียวต่อคัน)
```
ฝ่ายขาย/ธุรการ:
1. เพิ่ม/เลือกคนขับใน /drivers  → กรอกครบ: บัตร ปชช., วันเกิด, ที่อยู่, บัญชีธนาคาร
2. เพิ่ม/เลือกรถใน /vehicles    → กรอกครบ: ยี่ห้อ รุ่น เลขตัวถัง เลขเครื่อง วันจดทะเบียน
3. ตรวจราคาใน /price-list       → ดาวน์/งวด/ไฟแนนซ์
4. สร้างสัญญา /contracts/new    → ระบบ auto-fill จาก 1-3 + เลขสัญญาอัตโนมัติ
5. เช็ค checklist เขียว "ข้อมูลครบถ้วน" → พิมพ์สัญญา PDF ให้เซ็น
6. อัปโหลดสัญญาที่เซ็นแล้ว (saleContractUrl) + ตั้งโปรโมชั่น (promo_config)
```
**Definition of Done:** สัญญา active + checklist เขียว + ไฟล์เซ็นแล้วแนบ + โปรฯ ตั้งครบ

### 4.2 งานรายวัน/รายสัปดาห์
```
- Import เที่ยววิ่ง (/trips/import) — จาก แพล้นท์/GPS
- Import การเบิกอะไหล่ (/import stock-movement) — จากคลัง
- บันทึกใบรับสภาพหนี้เมื่อเกิดงานซ่อมที่คนขับต้องรับผิด
- แท็บ "จับคู่" ใน /vehicle-cost: จับ MR ↔ รายการเบิก → ติ๊กหักโปรซ่อม/PM → บันทึก
  (ยอดโปรฯ ตัดจริงทันที เห็นตรงกันทุกหน้า — เสร็จแล้ว 5 ก.ค.)
- เคสอุบัติเหตุ → เปิดเคส NC/AC (โมดูลใหม่ ดูข้อ 5.1)
```

### 4.3 รอบปิดเดือน (สำคัญที่สุด — เสนอทำเป็น Wizard)
```
วันที่ 1-3 ของเดือนถัดไป:
① ตรวจเที่ยววิ่งครบ         → ยอดรายได้ต่อสัญญา
② รวบรายการหักอัตโนมัติ:
   - ค่างวดรถ + งวดดาวน์     (installment_schedule / contract)
   - ภาษี/ประกัน/พรบ รายเดือน (contract.monthlyInsuranceFee ฯลฯ)
   - ค่า GPS                 (gps_config.monthlyFee)
   - ค่าน้ำมัน               (fuel_records)
   - ค่าซ่อม/อะไหล่ส่วนคนขับ  (debt_acceptances → งวดรายเดือน)
   - ยอดหักจากเคส NC/AC      (โมดูลใหม่)
   - รายการปรับปรุงอื่น       (monthly_adjustments)
③ รัน payroll engine → ตรวจ payslip → แก้ที่ต้นทางถ้าผิด → รันใหม่
④ อนุมัติ (role: admin) → export จ่ายเงิน + พิมพ์สลิปทั้งชุด
⑤ ปิดเดือน (month_status = closed) → ล็อกทุก transaction ของเดือนนั้น
```

## 5. สิ่งที่ต้องสร้าง/ปรับ (Gap → Action)

### 5.1 โมดูลใหม่: เคสเหตุการณ์ NC/AC (อุบัติเหตุ–เคลมประกัน) 🔴
> ปัจจุบันไม่มีที่เก็บเหตุการณ์อุบัติเหตุ/ความเสียหายเป็นเคส — มีแต่ยอดปลายทางใน debt
- collection ใหม่ `incident_cases`: เลขเคส, วันที่เกิดเหตุ, ทะเบียน/สัญญา/คนขับ, ประเภท (**AC = เคลมประกัน / NC = ไม่เข้าประกัน** — *ต้องยืนยันนิยามกับทีม*), คำอธิบาย+รูป, ค่าเสียหายรวม, ประกันจ่าย, ค่าเสียหายส่วนแรก, ส่วนที่คนขับรับผิด, สถานะ (เปิด→ประเมิน→เคลม→ปิด), ผูกใบรับสภาพหนี้เมื่อคนขับต้องผ่อนชดใช้
- หน้า /incidents: รายการเคส + timeline ต่อเคส + สรุปต่อคัน/ต่อคนขับ
- เชื่อม payroll: ยอดหักจากเคส (ผ่าน debt_acceptances เดิม) — engine รองรับอยู่แล้ว

### 5.2 ภาษี/ประกัน/พรบ — ทำเป็นระบบรอบต่ออายุ 🟡
- หน้า /tax-insurance: ตารางทุกคัน + วันหมดอายุ (ภาษี, พรบ, ประกันภัย, ตรวจสภาพ) + สถานะใกล้ครบ (แจ้งเตือน 60/30 วัน — มี alerts อยู่แล้ว ต่อยอดได้)
- บันทึกการจ่ายจริงต่อปี → ระบบแตกเป็นยอดหักรายเดือน (taxMonthlyInstallment) เข้า payroll อัตโนมัติ

### 5.3 Master data ให้ครบ (จำเป็นต่อสัญญา PDF และรายงาน) 🟡
- Campaign กรอกข้อมูล: drivers (บัตร ปชช./วันเกิด/ที่อยู่ — 100 คน) + vehicle_master (สเปครถ — เอาจากเล่มทะเบียน)
- Import จากไฟล์ ขนส่งฯ/เล่มทะเบียน ถ้ามี Excel อยู่แล้ว (ทำ template ให้)
- Validation: สัญญาใหม่ต้อง checklist เขียวก่อน active

### 5.4 รวมระบบโปรโมชั่น 2 ระบบเป็นหนึ่ง 🟡
- ปัจจุบัน: `promotion_entries` (เปิด/ระงับสิทธิ์ ต่อคัน) กับ `promo_config` (วงเงิน) แยกกัน
- เสนอ: `promo_config` เป็นแหล่งวงเงินเดียว + entries เป็นแค่สวิตช์เปิด/ระงับ + เหตุผล (ผูกกันด้วยทะเบียน — เริ่มเชื่อมแล้วในหน้า list 5 ก.ค.)
- vehicle_costs (แท็บแรก) นิยามให้ชัด: เป็น "ค่าใช้จ่ายทั่วไปที่ไม่หักโปรฯ" — ใส่ label บอกบนหน้า

### 5.5 ปิดเดือน + Audit 🟡
- month_status = closed → API ทุกตัว reject การเขียนย้อนเดือนที่ปิด (middleware ตรวจ date/month บน POST/PUT/PATCH/DELETE)
- เก็บ audit log (ใคร แก้อะไร เมื่อไหร่) — collection `audit_log` เขียนจาก API wrapper เดียว

### 5.6 รายงานผู้บริหาร 🟢 (ต่อยอดง่าย)
- P&L ต่อคัน/ต่อเดือน: รายได้เที่ยว − (ค่างวด+ซ่อม+โปรฯ ที่บริษัทออก+ภาษีประกัน+GPS) → กำไรต่อคัน
- Dashboard ความเสี่ยง: ค้างงวด, งบซ่อมใกล้หมด, ภาษีใกล้ขาด, เคส NC/AC เปิดค้าง

## 6. Roadmap เสนอ (4 เฟส)

| เฟส | ระยะ | ขอบเขต | ผลลัพธ์ |
|---|---|---|---|
| **1. Data Foundation** | 1-2 สัปดาห์ | ✅ sync โปรฯ↔ค่าใช้จ่าย (เสร็จ 5 ก.ค.) · กรอก master data ครบ · รวมระบบโปรฯ · normalize plate ทุกจุด | ตัวเลขเชื่อถือได้ทั้งระบบ |
| **2. Monthly Close** | 2-3 สัปดาห์ | Wizard ปิดเดือน ①-⑤ · ภาษี/ประกันรายเดือนอัตโนมัติ · ล็อกเดือน + audit log | ปิดเดือนได้ใน 1 วัน แทน Excel |
| **3. NC/AC + งานเคส** | 2 สัปดาห์ | โมดูล incident_cases + เชื่อม debt/payroll + รูปแนบ (DO Spaces มีแล้ว) | ทุกเหตุการณ์มีเคส ติดตามได้ |
| **4. Intelligence** | ต่อเนื่อง | P&L ต่อคัน · dashboard ความเสี่ยง · แจ้งเตือน LINE/อีเมล · (อนาคต) พอร์ทัลคนขับดูสลิปเอง | ใช้ตัดสินใจธุรกิจ |

## 8. ข้อเสนอเชิงเทคนิค (สั้น)
- คง Next.js + MongoDB เดิม — สเกลพอสำหรับ 100-500 คัน
- ย้าย secret ออกจาก script python → อ่าน .env (มี URI hardcode ในหลาย script)
- เพิ่ม index: stock_movements{mr, promoType}, trips{contractCode, month}, incident_cases{contractCode, status}
- ทำ staging DB แยกก่อนเล่นข้อมูลจริง เมื่อทีมเริ่มใช้งานประจำ
- Backup MongoDB รายวัน (DigitalOcean มี snapshot — ตั้ง schedule)
